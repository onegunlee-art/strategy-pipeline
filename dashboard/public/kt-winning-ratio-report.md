# KT 수주전략 Winning Ratio 자동화 툴 — 데모 개요 보고서

> 작성일: 2026년 6월 | 목적: 수주 확률 자동화 엔진의 컨셉·수학적 타당성 검증

---

## 0. 한 줄 요약

> **실제 Winning Ratio 대시보드는 글로벌 뉴스를 쓰지 않는다.**
> 이번 데모는 "정보 간 일치·충돌에서 확률을 산출하는 엔진"이 제대로 작동하는지를
> **수주 데이터 대신 글로벌 뉴스로 먼저 검증**한 것이다.
> 엔진은 정보의 *내용*이 아니라 *구조*만 입력받기 때문에,
> 승인 후 입력을 수주 데이터로 교체하면 **수식 변경 없이** 그대로 작동한다.

---

## 1. 왜 만들었는가

### Pain Point
KT 수주전략팀의 입찰 승률 판단은 현재 **담당자의 경험치와 직관**에 의존한다.
- 판단 기준이 개인마다 달라 **일관성 부재**
- 복수 입찰 건의 우선순위 비교가 어렵고 **의사결정 지연**
- 승패 원인이 축적되지 않아 **학습 효과 미흡**

### 목표
> 상충하는 정보의 흐름 속에서 **수주 성공 확률을 자동·정량 산출**하고,
> 확률을 높이는 전략을 즉시 제시하는 **살아있는 대시보드** 구축

---

## 2. 핵심 통찰 — 수주와 뉴스는 "같은 수학 문제"다

수주 승률 예측의 본질은 다음과 같다:

> **불완전하고 서로 상충하는 정보가 매일 들어오는 가운데,
> 미래의 이분형 결과(수주 성공/실패)의 확률을 추정하는 문제**

글로벌 뉴스 분석도 **정확히 동일한 구조**다. 미래 사건(예: 종전 여부)을
매일 갱신되는 상충된 보도로부터 확률 추정한다. 이 동형성(isomorphism)이 데모의 근거다.

| 수주 환경의 특성 | 글로벌 뉴스의 동형 특성 |
|---|---|
| 매일 입찰 변수·경쟁 구도 변동 | 매일 새 기사 생성 |
| 영업·기술·경쟁사 견해 **상충** | 매체별 상충 견해 (FA vs FT vs Economist) |
| 단일 정답 없음 → 확률 추정 | 미래 사건 → 확률 추정 |
| 정보 누적 → 판단 수렴 | 신호 누적 → Posterior 수렴 |

→ **"매일 생성 + 다양한 견해의 충돌"** 이라는 수주 환경의 본질을 뉴스가 그대로 재현한다.
   즉 뉴스는 수주 데이터의 **완벽한 대리 시험장(proxy testbed)** 이다.

## 3. 글로벌 뉴스로 무엇을 검증했나 — 수학적 트리거

### 핵심: 확률은 "내용"이 아니라 "일치/충돌 지점"에서 발화된다

엔진은 기사 원문을 읽어 확률을 매기는 게 아니다.
복수 소스를 **N개 드라이버 축**으로 정규화한 뒤, 소스 간 **일치(alignment)·충돌(conflict)**
을 정량화하여 확률을 계산한다.

- **Alignment point**: 복수 소스가 같은 방향 → 해당 드라이버 신뢰도↑, 확률 기여 강화
- **Conflict point**: 소스가 반대 방향 → 불확실성↑ (분포 표준편차 확대), 순델타 상쇄

### 확률 수식 (실제 구현)

**① 드라이버 기여도** — invert 축은 뒤집어 정규화
```
contribution(d) = d.invert ? (10 − score_d) : score_d        (0 ~ 10)
```

**② Prior 확률** — 기여도 평균 × 10
```
P₀ = clamp(5, 95, round( mean(contribution) × 10 ))
```

**③ Bayesian Posterior** — 일치/충돌 신호를 가중 델타로 누적
```
score'_d = clamp(0, 10,  score_d + Σᵢ(δᵢ,d · wᵢ) / (κ + Σᵢ wᵢ))

  δᵢ,d = 시그널 i 의 드라이버 d 방향 델타 (일치 +, 충돌 −)
  wᵢ   = 소스/투표자 가중치 (역할별)
  κ    = PRIOR_STRENGTH = 10   (AI 사전판단을 10표로 취급)

P = clamp(5, 95, round( mean(contribution(score')) × 10 ))
```

**④ 신뢰구간 수렴** — 신호가 쌓일수록 분포가 좁아진다
```
σ = σ₀ / √(1 + n/3)        (n = 누적 신호 수)
```

### 이 수식의 결정적 성질 — **도메인 불변(domain-invariant)**

위 어느 수식에도 **"뉴스"라는 단어가 박혀 있지 않다.**
입력은 오직 **(드라이버 점수, 일치/충돌 가중 델타)** 뿐이다.
초기엔 AI 사전판단(κ=10표)이 지배하고, 신호가 누적될수록 집단 판단으로 수렴한다
(1표의 최대 확률 이동 ≈ 0.9%로 과민반응 방지).

→ **입력의 출처가 뉴스든 수주 데이터든, 엔진은 같은 방식으로 작동한다.**

### 검증 결과
데이터 없이도 **확률 트리거·Bayesian 수렴·투표 메커니즘·리포트 생성 전체 동작 확인 완료**

## 4. 왜 그대로 수주에 적용되는가

실제 대시보드는 뉴스 수집부를 떼어내고, **동일한 확률 엔진**에
수주 데이터를 입력으로 연결하기만 하면 된다. **수식·로직 변경 0.**

| 구성 요소 | 데모 (뉴스) | 실 운영 (수주) | 엔진 변경 |
|---|---|---|---|
| 입력 소스 | 글로벌 뉴스 기사 | 수실주 보고서·경쟁사 DB·발주사 정보 | **없음** |
| 드라이버 5축 | 외교·군사·경제·국내·외부 | 기술력·가격·관계·레퍼런스·리스크 | 라벨만 교체 |
| 일치/충돌 신호 | 매체 간 보도 상충 | 영업·기술·경쟁사 인텔리전스 상충 | **없음** |
| 투표자 | 데모 참여자 | 수주팀·BD팀·임원 | **없음** |
| 확률 의미 | 사건 발생 가능성 | **수주 성공 확률** | **없음** |

→ Bayesian 업데이트 · 일치/충돌 트리거 · 투표 가중 · 리포트 생성 **코드 재사용률 90%+**

---

## 5. 승인 후 로드맵 — "살아있는" Winning Ratio 대시보드

**Phase 1 — 실 DB 연결** (1~2개월)
1. 수실주 DB 연결: 과거 5년 입찰 결과 → 드라이버 가중치·κ 보정 학습
2. 경쟁사·발주사 정보 파이프라인 → 일치/충돌 신호 자동 생성

**Phase 2 — Weekly 시장조사 레포트 발간** (정례화)
3. 매주 시장·경쟁 동향 자동 수집 → 진행 입찰 건 확률 갱신
4. 변동 사유(어느 드라이버가 왜 움직였는지)까지 리포트로 발간

**Phase 3 — 집단지성 + 살아있는 대시보드**
5. 수주팀 실시간 투표 → 현장 인텔리전스 즉시 반영
6. 승패 피드백 루프 → 드라이버 가중치 자동 재보정 (정확도 지속 향상)
7. 임원 대시보드: 전체 포트폴리오 수주 확률 히트맵

→ 데모의 "정지된 스냅샷"이, 매주 갱신되는 **정확하고 살아있는 수주 확률 대시보드**로 전환된다.
